# Yarukoto 開発フローガイド & プロンプト集 v1

> **このファイルについて**
> バイブコーディングでの機能改修を「プロンプトを順番に投げるだけ」で回すためのガイド。
> 人間がやるのは「課題を見つける」「プロンプトを投げる」「実機で確認する」「リリースする」の4つだけ。
>
> **エージェント調整指示は末尾に記載。**

---

## 全体フロー概要

```
フェーズ1 【人間】課題の抽出
    ↓ 成果物：課題メモ（テキストでOK）
フェーズ2 【AI】課題の整理・ISSUES.md更新
    ↓ 成果物：ISSUES.md
フェーズ3 【AI】修正ロードマップ更新
    ↓ 成果物：ROADMAP.md
フェーズ4 【AI】実装 → 検証（枝番単位で繰り返し）
    ↓ 成果物：コード変更、qa-report.md、WORK-LOG.md
フェーズ5 【人間】動作確認
    ↓ 成果物：追加の課題メモ（あれば）
フェーズ6 【人間】ビルド指示
フェーズ7 【AI】ビルド・リリースノート作成
    ↓ 成果物：実行ファイル、RELEASE_NOTES.md
フェーズ8 【人間】最終確認 → GitHubリリース
    ※ 問題があればフェーズ5に戻る
```

### コンテキストリセットの原則

| フェーズ | リセットタイミング | 備考 |
|---|---|---|
| フェーズ2 | フェーズ開始時に新規チャット | — |
| フェーズ3 | フェーズ開始時に新規チャット | — |
| フェーズ4（実装） | **枝番ごと**に新規チャット | 1枝番 = 1チャット |
| フェーズ4（検証） | **検証STEPごと**に新規チャット | QAプロンプト集に準拠 |
| フェーズ7 | フェーズ開始時に新規チャット | — |

### 各チャットでAIに最初に読ませるもの

どのフェーズでも、チャットの冒頭で以下を指示する：

```
プロジェクトルート直下の CLAUDE.md を読んでプロジェクトの概要を把握すること。
```

> **Note**: Claude Code は `CLAUDE.md` を自動的にコンテキストに読み込むため、
> Claude Code 利用時にはこの指示は重複する。ただし Antigravity 等の他エージェントでは
> 自動読み込みされないため、プロンプトへの明記はそのまま維持すること。

これに加えて、フェーズごとに追加で読ませるファイルがある（各プロンプトに記載済み）。

### チャットが途中で切れた場合の復帰手順

AIのエラーやネットワーク断でチャットが途中終了した場合、WORK-LOG.md が更新されないまま
作業が宙に浮くことがある。その場合は以下の **復帰プロンプト** を新しいチャットで投げる。

```
プロジェクトルート直下の CLAUDE.md を読んでプロジェクトの概要を把握すること。
続けて以下のファイルを順に読むこと：
- WORK-LOG.md
- ROADMAP.md
- qa-report.md（存在する場合）

さらに、直近のgit履歴を確認すること（git log --oneline -10）。

上記の情報から、現在の作業状況を判断し、以下を報告すること：
- 現在どのフェーズ・枝番の作業中か
- 最後に完了した作業は何か
- 次にやるべき作業は何か（具体的なプロンプト番号で示すこと）
- WORK-LOG.md の内容と git log に矛盾がある場合はその旨

コードの修正は行わないこと。状況の報告のみ行うこと。
```

AIの報告を見て、該当するプロンプトから作業を再開する。

## 管理ファイル一覧と役割

| ファイル | 役割 | 誰が書く | いつ読む |
|---|---|---|---|
| `CLAUDE.md` | プロジェクト概要・技術スタック・コーディング規約・ビルド手順 | AI（初回）、人間が随時調整 | 毎チャット冒頭 |
| `ISSUES.md` | 課題の一覧と状態管理 | AI（フェーズ2） | フェーズ3, 4, 7 |
| `ROADMAP.md` | バージョン別の修正計画（枝番・検証計画込み） | AI（フェーズ3） | フェーズ4の各チャット冒頭 |
| `WORK-LOG.md` | 直前チャットの作業記録 | AI（各チャット終了時） | 次のチャットの冒頭 |
| `AI_CHANGELOG.md` | AIエージェントによる変更履歴（全セッション通算） | AI（各セッション終了時） | 必要に応じて参照 |
| `qa-report.md` | 検証結果の蓄積 | AI（フェーズ4検証） | フェーズ4修正、フェーズ5 |
| `RELEASE_NOTES.md` | リリースノート | AI（フェーズ7） | フェーズ8 |

> **Note**: `AI_CHANGELOG.md` はプロジェクト全体の変更履歴を長期保存するファイル。
> `WORK-LOG.md` はチャット間の短期引き継ぎ用で、直近3件のみ保持する。用途が異なるため両方維持する。

> **Note**: v1.1.0 の検証時には `VERIFICATION_BUGS.md` が `qa-report.md` の前身として使われていた。
> v1.2.0 以降は本ガイドの `qa-report.md` フォーマットに統一する。

## Gitコミットの運用ルール

`CLAUDE.md` に定義済みの通り、AIエージェントは**キリの良いタイミングで自律的に `git add .` と `git commit` を実行**し、履歴を残すこと。ユーザーにコミットの許可を取る必要はない。

各フェーズのプロンプトにはコミット指示を明示していないが、実装や修正が一段落したタイミングで自動的にコミットすること。

## WORK-LOG.md のフォーマットと運用ルール

コンテキストリセットをまたいで作業の連続性を保つための核となるファイル。

### 運用ルール

- 各チャットの終了時にAIが「最新の作業」セクションを更新する
- **「過去の作業」は直近3件まで保持する。4件目以降は削除する。**
  - ファイルの肥大化を防ぎ、AIが読み込むトークンを節約するため
  - 詳細な作業履歴が必要な場合は git log や `AI_CHANGELOG.md`、`qa-report.md` を参照する
- 次のチャットのAIが最初に読むことで「前回どこまで終わったか」を把握する

### フォーマット

```markdown
# Work Log

## 最新の作業（YYYY-MM-DD HH:MM）

- **フェーズ**: フェーズ4 実装
- **対象バージョン**: v1.2.0
- **対象枝番**: 2-1（IMP-3, IMP-7）
- **ステータス**: ✅ 完了 / 🔧 作業中 / ⏸️ 中断
- **やったこと**:
  - （具体的な変更内容を箇条書き）
- **変更したファイル**:
  - `components/TaskList.js` — 変更概要
  - `app/settings/page.js` — 変更概要
- **次にやるべきこと**:
  - （次のチャットで着手すべき作業を明記）
- **注意事項・申し送り**:
  - （次のAIが知っておくべきこと。判断に迷ったこと、保留にしたことなど）

---

## 過去の作業（直近3件まで保持。4件目以降は削除すること）

### YYYY-MM-DD HH:MM — フェーズ4 実装 2-1（IMP-3, IMP-7）
- ステータス: ✅ 完了
- やったこと: ...
- 変更したファイル: ...

### YYYY-MM-DD HH:MM — フェーズ3 ロードマップ更新
- ステータス: ✅ 完了
- やったこと: ...

### YYYY-MM-DD HH:MM — フェーズ2 課題整理
- ステータス: ✅ 完了
- やったこと: ...
```

# フェーズ別プロンプト

## フェーズ1：課題の抽出（人間の作業）

**プロンプト不要。** 人間が使いながら気になったことをメモする。

メモの書き方ガイド：
- 形式は自由（箇条書き、走り書きでOK）
- 以下の情報があるとフェーズ2でAIが正確に整理できる：
  - **どの画面で**（例：タスク一覧画面、今日やるタスク画面）
  - **何をしたら**（例：タスクを削除しようとしたら）
  - **どうなった/どうしたい**（例：確認なしで消えた → 確認ダイアログほしい）
- 種別（バグ/改善/機能追加）の判断は不要。AIがやる。

### メモの例

```
- タスク一覧で削除押したら確認なしで消える。怖い
- 完了したタスクがいつまでも一覧に残って邪魔
- カテゴリの並び順を変えたい
- たまに保存ボタン押しても反応ない時がある
- 起動が前より遅くなった気がする
```

## フェーズ2：課題の整理・ISSUES.md更新

**新しいチャットを開いて以下を投げる。**
課題メモの部分は、自分のメモをそのまま貼り付ける。
同じチャット内で「分析→人間の確認→書き込み」の2ステップで進める。

### プロンプト2-A：課題の分析・レビュー依頼

```
プロジェクトルート直下の CLAUDE.md を読んでプロジェクトの概要を把握すること。
続けて ISSUES.md を読むこと。

以下は、現行バージョンを使って気づいた課題のメモである。

---
（ここに自分の課題メモを貼り付ける）
---

上記の課題メモを整理・分析すること。ただし ISSUES.md にはまだ書き込まないこと。

作業手順：
1. 各課題を以下のように分類する
   - BUG: 意図しない動作、クラッシュ、データ不整合
   - IMP: 既存機能の使い勝手改善、UI/UX改善
   - FEAT: 新しい機能の追加
2. 重要度を判定する
   - 高: クラッシュ、データ損失、主要機能が使えない
   - 中: 使い勝手に明確な問題がある、頻繁に遭遇する
   - 低: 軽微な見た目の問題、稀にしか遭遇しない
3. 既存のISSUES.mdの課題と重複がないか確認する
   - 重複する場合は既存のIDを維持し、必要に応じて説明を更新する
   - 新規の場合はIDを採番する（BUG-N, IMP-N, FEAT-N）

分析結果を以下の形式で報告すること（ISSUES.md への書き込みはまだ行わない）：
- 新規追加する課題の一覧（ID、概要、分類、重要度）をテーブル形式で提示
- 既存課題との重複があった場合はその旨
- 分類や重要度の判断に迷ったものがあればその理由
- 修正による懸念点（影響範囲、他機能への波及、仕様変更を伴うもの）があれば

人間の確認が取れるまで ISSUES.md の更新は行わないこと。
```

**人間がやること:** AIの分析結果を確認し、修正指示があれば伝える。問題なければ「OK」と返す。

### プロンプト2-B：ISSUES.md への書き込み（同じチャット内で続けて投げる）

```
確認が取れたので、分析結果を ISSUES.md に反映すること。

- フォーマットは本ガイドの「ISSUES.md のフォーマット（参考）」に従うこと
  （テーブル形式 + 仕様詳細は <details> で折りたたみ）
- 既存の散文形式の項目がある場合はテーブル形式に変換すること
- 先ほどの確認で修正指示があった場合は、それを反映した上で書き込むこと

更新完了後、変更箇所のサマリーを報告すること。
```

## フェーズ3：修正ロードマップ更新

**新しいチャットを開いて以下を投げる。**

```
プロジェクトルート直下の CLAUDE.md を読んでプロジェクトの概要を把握すること。
続けて ISSUES.md と ROADMAP.md を読むこと。

ISSUES.md の未着手課題（🔴）を踏まえて、ROADMAP.md を更新すること。

作業手順：

1. 未着手課題を確認し、以下の基準でバージョンに振り分ける
   - 重要度「高」のBUG → 最も近いバージョンに入れる
   - 重要度「高」のIMP → 同上
   - 重要度「中」 → 内容と量を見て適切なバージョンに配分する
   - 重要度「低」 → 余裕があれば入れる。無理に入れない
   - FEAT（機能追加） → 原則としてメジャーバージョンアップ（vX.0.0）に配置

2. 各バージョン内で枝番を構成する
   - 1つの枝番 = 1回のAIチャットで実装する単位
   - 枝番の記法は既存の ROADMAP.md に合わせる（例: 2-1, 2-2, 3-1）
   - 枝番の構成ルール：
     a. 関連する課題はまとめる（同じ画面、同じ機能に関するもの）
     b. ただし1枝番あたり最大3課題まで（多すぎると品質が落ちる）
     c. 難易度が高い課題は単独で1枝番にする
     d. BUGとIMPは混ぜてよいが、FEATは単独の枝番にする
   - 枝番の順序：依存関係がある場合はそれに従う。なければBUG優先

3. 各枝番に検証計画を設定する
   - 以下の基準で検証STEPを選ぶ：
     a. 基本セット（全枝番共通）: STEP 1, 2, 3
     b. DB変更を伴う場合: 上記 + STEP 4
     c. UI文言やスタイルの変更を伴う場合: 上記 + STEP 5
     d. バージョンの最終枝番: STEP 1〜5 フルセット + STEP 8 + STEP R
     e. メジャーバージョンアップの最終枝番: STEP 1〜7 フルセット + STEP 8 + STEP R
   - 各枝番の実装後に STEP R（リグレッション）を実施する

4. ROADMAP.md を更新して保存する

更新後、以下を報告すること：
- 各バージョンの枝番構成の一覧（枝番番号、対象課題ID、検証計画）
- 1枝番あたりの作業量の見積もり（軽い/普通/重い）
- 振り分けや順序で迷った判断があれば相談すること

ISSUES.md は更新しないこと（ステータスの更新はフェーズ4で行う）。
```

## フェーズ4：実装・検証

> **フェーズ4の全体像**
> ROADMAPの枝番を1つずつ消化していく。
> 1枝番につき「実装 → 検証 → 修正 → リグレッション」のサイクルを回す。
> 各ステップは別チャットで実行する。

### フェーズ4 内のチャット構成

```
枝番ごとに以下を繰り返す：

チャットA: 実装
チャットB: 検証 STEP 0 + STEP 1
チャットC: 検証 STEP 0 + STEP 2
チャットD: 検証 STEP 0 + STEP 3
 （ROADMAPで指定された検証STEPの数だけ続く）
チャットE: 修正 STEP 0 + STEP 8
チャットF: STEP R準備（チャットE内で実施。別チャット不要）
チャットG: リグレッション STEP 0 + STEP R
チャットH: 仕上げ（ISSUES.md更新、WORK-LOG更新）
```

### どのプロンプトを投げればいいかの早見表

| やること | 投げるプロンプト | 備考 |
|---|---|---|
| 枝番の実装を始める | プロンプト4-A | 1枝番につき1回 |
| 検証する | プロンプト4-B | ROADMAPに書かれたSTEPの数だけ繰り返す |
| 検証で見つかった問題を修正する | プロンプト4-C | qa-report.mdにNGがあれば実施 |
| 修正後のリグレッション準備 | プロンプト4-D | 4-Cと同じチャット内で続けて投げる |
| リグレッションテスト実行 | プロンプト4-E | 新しいチャットで実施 |
| 枝番の完了処理 | プロンプト4-F | ISSUES.md更新 + WORK-LOG更新 |
| qa-report.mdにまだNGが残っている | プロンプト4-Cに戻る | NGが0になるまで繰り返し |

### フェーズ4 の判断フローチャート

> 迷ったらこのフローチャートではなく、末尾の **クイックリファレンス** を見ること。
> 「今の状況 → 次にやること」が一目でわかる。

```
枝番の実装は済んだ？
├─ No → プロンプト4-A（実装）
└─ Yes
    ↓
  ROADMAPで指定された検証STEPはすべて実施した？
  ├─ No → プロンプト4-B（検証：次のSTEP）
  └─ Yes
      ↓
    qa-report.md にNG項目がある？
    ├─ No → プロンプト4-F（完了処理）
    └─ Yes
        ↓
      プロンプト4-C（修正）
        ↓
      同じチャットで プロンプト4-D（リグレッション準備）
        ↓
      プロンプト4-E（リグレッション）
        ↓
      リグレッションでNGが出た？
      ├─ No → プロンプト4-F（完了処理）
      └─ Yes → プロンプト4-C に戻る（3回超えたら「打ち切りルール」参照）
```

### プロンプト4-A：実装

**新しいチャットを開いて投げる。**

```
プロジェクトルート直下の CLAUDE.md を読んでプロジェクトの概要を把握すること。
続けて以下のファイルを順に読むこと：
- ISSUES.md
- ROADMAP.md
- WORK-LOG.md

上記を読んだ上で、以下の作業を行うこと。

■ 今回の作業対象
バージョン: v___
```

**新しいチャットを開いて投げる。**
共通ヘッダー → コピーした【変更サマリー】 → 以下のSTEP R本体 の順で貼る。

```
【STEP R：リグレッションテスト】

上記の【変更サマリー】に基づいて、以下の2段階でリグレッションテストを行うこと。

■ 第1段階：変更箇所の直接テスト
変更した機能そのものが正常に動作するか確認する。

■ 第2段階：影響範囲の特定とテスト
1. 【変更サマリー】の「影響が想定される箇所」を起点に、コード上でさらに参照元を洗い出す
2. 洗い出した箇所の機能が変更前と同様に動作するか確認する
3. データの流れを追い、間接的に影響を受ける画面や処理があれば、それも確認する

報告形式：
| # | テスト区分（直接/影響範囲） | 機能名 | 操作内容 | 期待結果 | 実際の結果 | OK/NG |

コードの修正はしないこと。

結果はプロジェクトルート直下の qa-report.md の末尾に追記すること。
セクション見出しは「## STEP R：リグレッションテスト（vX.X.X 枝番N-M YYYY-MM-DD）」とすること。

【記載ルール】
このファイルは別チャットのAIが読んで修正作業を行う。
そのAIがこのファイルだけで修正箇所を特定できるレベルの詳細さで書くこと。
- NG項目には必ず以下をすべて含めること：
  - 該当ファイル名と行番号
  - 再現手順
  - 期待される挙動と実際の挙動の差分
  - 原因の推定
  - 今回の変更との因果関係（この変更が原因か、元から壊れていたか）
- OK項目は表の1行で十分。
- 影響範囲の洗い出し結果（どのファイル・関数を確認対象としたか）も記載すること。
```

### 補足：修正→検証の繰り返しが3回を超えた場合

修正とリグレッションを3回繰り返してもNGが解消しない場合は、
根本的な設計の問題がある可能性がある。

対処：
1. qa-report.md の残存NG項目を確認する
2. 残存NGの内容を課題メモとして、フェーズ2（課題の整理）に戻す
3. 該当の枝番は ROADMAP.md で「⏸️ 保留」とする
4. ISSUES.md に新規課題として登録し、次バージョン以降で対応を検討する

### 補足：リリース前検証

バージョンの全枝番が完了した後、ROADMAPの「リリース前検証」で指定されたSTEPを実施する。

リリース前検証でNGが出た場合は、プロンプト4-C → 4-D → 4-E の修正サイクルを回す。

## ISSUES.md フォーマット移行の提案

**現状**: 散文形式（ヘッダー+箇条書き、仕様詳細を直接記述）
**推奨**: テーブル形式（一覧性向上）+ `<details>` で仕様詳細を折りたたみ

- **メリット**: 一覧性が上がり、ステータス管理がしやすくなる。AIが機械的に読み取りやすい。
- **デメリット**: 既存の詳細な仕様記述を移行する手間がかかる。
- **移行タイミング**: v1.2.0 のフェーズ2で ISSUES.md を更新する際に、AIにフォーマット変換を指示する。
