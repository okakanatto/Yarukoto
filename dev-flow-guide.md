# Yarukoto 開発フローガイド & プロンプト集 v1

> **このファイルについて**
> バイブコーディングでの機能改修を「プロンプトを順番に投げるだけ」で回すためのガイド。
> 人間がやるのは「課題を見つける」「プロンプトを投げる」「実機で確認する」「リリースする」の4つだけ。
>
> **エージェント調整指示は末尾に記載。**

---

## 全体フロー概要

```
フェーズ1 【人間】課題の抽出
    ↓ 成果物：課題メモ（テキストでOK）
フェーズ2 【AI】課題の整理・ISSUES.md更新
    ↓ 成果物：ISSUES.md
フェーズ3 【AI】修正ロードマップ更新
    ↓ 成果物：ROADMAP.md
フェーズ4 【AI】実装 → 検証（枝番単位で繰り返し）
    ↓ 成果物：コード変更、qa-report.md、WORK-LOG.md
フェーズ5 【人間】動作確認
    ↓ 成果物：追加の課題メモ（あれば）
フェーズ6 【人間】ビルド指示
フェーズ7 【AI】ビルド・リリースノート作成
    ↓ 成果物：実行ファイル、RELEASE_NOTES.md
フェーズ8 【人間】最終確認 → GitHubリリース → 【AI】後片付け
    ※ 問題があればフェーズ5に戻る
```

### コンテキストリセットの原則

| フェーズ | リセットタイミング | 備考 |
|---|---|---|
| フェーズ2 | フェーズ開始時に新規チャット | 同一チャット内で2-A→確認→2-B |
| フェーズ3 | フェーズ開始時に新規チャット | 同一チャット内で3-A→確認→3-B |
| フェーズ4（実装） | **枝番ごと**に新規チャット | 1枝番 = 1チャット |
| フェーズ4（検証） | **検証STEPごと**に新規チャット | STEP A, B, C それぞれ別チャット |
| フェーズ4（修正） | 修正ごとに新規チャット | 4-C + 4-D を同一チャット内で実施 |
| フェーズ4（リグレッション） | 新規チャット | NG0件なら完了処理まで同一チャット内で実施 |
| フェーズ7 | フェーズ開始時に新規チャット | — |
| フェーズ8（後片付け） | 新規チャット | — |

### 各チャットでAIに最初に読ませるもの

どのフェーズでも、チャットの冒頭で以下を指示する：

```
プロジェクトルート直下の CLAUDE.md を読んでプロジェクトの概要を把握すること。
```

> **Note**: Claude Code は `CLAUDE.md` を自動的にコンテキストに読み込むため、
> Claude Code 利用時にはこの指示は重複する。ただし Antigravity 等の他エージェントでは
> 自動読み込みされないため、プロンプトへの明記はそのまま維持すること。

これに加えて、フェーズごとに追加で読ませるファイルがある（各プロンプトに記載済み）。

### チャットが途中で切れた場合の復帰手順

AIのエラーやネットワーク断でチャットが途中終了した場合、WORK-LOG.md が更新されないまま
作業が宙に浮くことがある。その場合は以下の **復帰プロンプト** を新しいチャットで投げる。

```
プロジェクトルート直下の CLAUDE.md を読んでプロジェクトの概要を把握すること。
続けて以下のファイルを順に読むこと：
- WORK-LOG.md
- ROADMAP.md
- qa-report.md（存在する場合）

さらに、直近のgit履歴を確認すること（git log --oneline -10）。

上記の情報から、現在の作業状況を判断し、以下を報告すること：
- 現在どのフェーズ・枝番の作業中か
- 最後に完了した作業は何か
- 次にやるべき作業は何か（具体的なプロンプト番号で示すこと）
- WORK-LOG.md の内容と git log に矛盾がある場合はその旨

コードの修正は行わないこと。状況の報告のみ行うこと。
```

AIの報告を見て、該当するプロンプトから作業を再開する。

## 管理ファイル一覧と役割

| ファイル | 役割 | 誰が書く | いつ読む |
|---|---|---|---|
| `CLAUDE.md` | プロジェクト概要・技術スタック・コーディング規約・ビルド手順 | AI（初回）、人間が随時調整 | 毎チャット冒頭 |
| `ISSUES.md` | 課題の一覧と状態管理 | AI（フェーズ2, 8） | フェーズ3, 4, 7, 8 |
| `ROADMAP.md` | バージョン別の修正計画（枝番・検証計画込み） | AI（フェーズ3, 8） | フェーズ4の各チャット冒頭, 8 |
| `WORK-LOG.md` | 直前チャットの作業記録 | AI（各チャット終了時） | 次のチャットの冒頭 |
| `AI_CHANGELOG.md` | AIエージェントによる変更履歴（全セッション通算） | AI（各セッション終了時） | 必要に応じて参照 |
| `qa-report.md` | 検証結果の蓄積 | AI（フェーズ4検証） | フェーズ4修正、フェーズ5 |
| `RELEASE_NOTES.md` | リリースノート | AI（フェーズ7） | フェーズ8 |

> **Note**: `AI_CHANGELOG.md` はプロジェクト全体の変更履歴を長期保存するファイル。
> `WORK-LOG.md` はチャット間の短期引き継ぎ用で、直近3件のみ保持する。用途が異なるため両方維持する。

> **Note**: v1.1.0 の検証時には `VERIFICATION_BUGS.md` が `qa-report.md` の前身として使われていた。
> v1.2.0 以降は本ガイドの `qa-report.md` フォーマットに統一する。

## Gitコミットの運用ルール

`CLAUDE.md` に定義済みの通り、AIエージェントは**キリの良いタイミングで自律的に `git add .` と `git commit` を実行**し、履歴を残すこと。ユーザーにコミットの許可を取る必要はない。

各フェーズのプロンプトにはコミット指示を明示していないが、実装や修正が一段落したタイミングで自動的にコミットすること。

## WORK-LOG.md のフォーマットと運用ルール

コンテキストリセットをまたいで作業の連続性を保つための核となるファイル。

### 運用ルール

- 各チャットの終了時にAIが「最新の作業」セクションを更新する
- **「過去の作業」は直近3件まで保持する。4件目以降は削除する。**
  - ファイルの肥大化を防ぎ、AIが読み込むトークンを節約するため
  - 詳細な作業履歴が必要な場合は git log や `AI_CHANGELOG.md`、`qa-report.md` を参照する
- 次のチャットのAIが最初に読むことで「前回どこまで終わったか」を把握する

### フォーマット

```markdown
# Work Log

## 最新の作業（YYYY-MM-DD HH:MM）

- **フェーズ**: フェーズ4 実装
- **対象バージョン**: v1.2.0
- **対象枝番**: 2-1（IMP-3, IMP-7）
- **ステータス**: ✅ 完了 / 🔧 作業中 / ⏸️ 中断
- **やったこと**:
  - （具体的な変更内容を箇条書き）
- **変更したファイル**:
  - `components/TaskList.js` — 変更概要
  - `app/settings/page.js` — 変更概要
- **次にやるべきこと**:
  - （次のチャットで着手すべき作業を明記）
- **注意事項・申し送り**:
  - （次のAIが知っておくべきこと。判断に迷ったこと、保留にしたことなど）

---

## 過去の作業（直近3件まで保持。4件目以降は削除すること）

### YYYY-MM-DD HH:MM — フェーズ4 実装 2-1（IMP-3, IMP-7）
- ステータス: ✅ 完了
- やったこと: ...
- 変更したファイル: ...

### YYYY-MM-DD HH:MM — フェーズ3 ロードマップ更新
- ステータス: ✅ 完了
- やったこと: ...

### YYYY-MM-DD HH:MM — フェーズ2 課題整理
- ステータス: ✅ 完了
- やったこと: ...
```

# フェーズ別プロンプト

## フェーズ1：課題の抽出（人間の作業）

**プロンプト不要。** 人間が使いながら気になったことをメモする。

メモの書き方ガイド：
- 形式は自由（箇条書き、走り書きでOK）
- 以下の情報があるとフェーズ2でAIが正確に整理できる：
  - **どの画面で**（例：タスク一覧画面、今日やるタスク画面）
  - **何をしたら**（例：タスクを削除しようとしたら）
  - **どうなった/どうしたい**（例：確認なしで消えた → 確認ダイアログほしい）
- 種別（バグ/改善/機能追加）の判断は不要。AIがやる。

### メモの例

```
- タスク一覧で削除押したら確認なしで消える。怖い
- 完了したタスクがいつまでも一覧に残って邪魔
- カテゴリの並び順を変えたい
- たまに保存ボタン押しても反応ない時がある
- 起動が前より遅くなった気がする
```

## フェーズ2：課題の整理・ISSUES.md更新

**新しいチャットを開いて以下を投げる。**
課題メモの部分は、自分のメモをそのまま貼り付ける。
同じチャット内で「分析→人間の確認→書き込み」の流れで進める（プロンプト2-A → 確認 → 2-B）。

### プロンプト2-A：課題の分析・レビュー依頼

```
プロジェクトルート直下の CLAUDE.md を読んでプロジェクトの概要を把握すること。
続けて ISSUES.md を読むこと。

以下は、現行バージョンを使って気づいた課題のメモである。

---
（ここに自分の課題メモを貼り付ける）
---

上記の課題メモを整理・分析すること。ただし ISSUES.md にはまだ書き込まないこと。

作業手順：
1. 各課題を以下のように分類する
   - BUG: 意図しない動作、クラッシュ、データ不整合
   - IMP: 既存機能の使い勝手改善、UI/UX改善
   - ENH: 新しい機能の追加
2. 重要度を判定する
   - 高: クラッシュ、データ損失、主要機能が使えない
   - 中: 使い勝手に明確な問題がある、頻繁に遭遇する
   - 低: 軽微な見た目の問題、稀にしか遭遇しない
3. 既存のISSUES.mdの課題と重複がないか確認する
   - 重複する場合は既存のIDを維持し、必要に応じて説明を更新する
   - 新規の場合はIDを採番する（BUG-N, IMP-N, ENH-N）

分析結果を以下の形式で報告すること（ISSUES.md への書き込みはまだ行わない）：
- 新規追加する課題の一覧（ID、概要、分類、重要度）をテーブル形式で提示
- 既存課題との重複があった場合はその旨
- 分類や重要度の判断に迷ったものがあればその理由
- 修正による懸念点（影響範囲、他機能への波及、仕様変更を伴うもの）があれば

人間の確認が取れるまで ISSUES.md の更新は行わないこと。
```

**人間がやること:** AIの分析結果を確認し、修正指示があれば伝える。問題なければ「OK」と返す。

### プロンプト2-B：ISSUES.md への書き込み（同じチャット内で続けて投げる）

```
確認が取れたので、分析結果を ISSUES.md に反映すること。

- フォーマットは本ガイドの「ISSUES.md のフォーマット（参考）」に従うこと
  （テーブル形式 + 仕様詳細は <details> で折りたたみ）
- 既存の散文形式の項目がある場合はテーブル形式に変換すること
- 先ほどの確認で修正指示があった場合は、それを反映した上で書き込むこと

更新完了後、変更箇所のサマリーを報告すること。
```

## フェーズ3：修正ロードマップ更新

**新しいチャットを開いて以下を投げる。**
フェーズ2と同様、「提案→人間の確認→書き込み」の2ステップで進める。

### プロンプト3-A：ロードマップの提案

```
プロジェクトルート直下の CLAUDE.md を読んでプロジェクトの概要を把握すること。
続けて ISSUES.md と ROADMAP.md を読むこと。

ISSUES.md の未着手課題（🔴）を踏まえて、ROADMAP.md の更新案を作成すること。
ただし ROADMAP.md にはまだ書き込まないこと。

作業手順：

1. 未着手課題を確認し、以下の基準でバージョンに振り分ける
   - 重要度「高」のBUG → 最も近いバージョンに入れる
   - 重要度「高」のIMP → 同上
   - 重要度「中」 → 内容と量を見て適切なバージョンに配分する
   - 重要度「低」 → 余裕があれば入れる。無理に入れない
   - ENH（機能追加） → 原則としてメジャーバージョンアップ（vX.0.0）に配置

2. バージョンの粒度を以下のガイドラインで調整する
   - 1バージョンあたり枝番3〜5個を目安にする（多すぎるとリリースが遅延し、少なすぎると管理が煩雑）
   - 「重い」枝番が2つ以上あるバージョンは作らない（品質リスク）
   - 重要度「高」のBUGが含まれるバージョンは小さく切って早くリリースする
   - 既存ROADMAPに未着手のバージョンがある場合はそこへの追加を優先し、新バージョンの乱立を避ける

3. 各バージョン内で枝番を構成する
   - 1つの枝番 = 1回のAIチャットで実装する単位
   - 枝番の記法は既存の ROADMAP.md に合わせる（例: 2-1, 2-2, 3-1）
   - 枝番の構成ルール：
     a. 関連する課題はまとめる（同じ画面、同じ機能に関するもの）
     b. ただし1枝番あたり最大3課題まで（多すぎると品質が落ちる）
     c. 難易度が高い課題は単独で1枝番にする
     d. BUGとIMPは混ぜてよいが、ENHは単独の枝番にする
   - 枝番の順序：依存関係がある場合はそれに従う。なければBUG優先

4. 各枝番に検証計画を設定する
   - 以下の基準で検証STEPを選ぶ：
     a. 基本セット（全枝番共通）: STEP A + STEP R
     b. DB変更またはUI変更を伴う場合: STEP A + STEP B + STEP R
     c. バージョンの最終枝番: STEP A + STEP B + STEP R
     d. メジャーバージョンアップの最終枝番: STEP A + STEP B + STEP C + STEP R
   - 各STEPの内容：
     - STEP A: 機能検証（正常系・異常系・状態遷移テスト）
     - STEP B: 品質レビュー（エラーハンドリング・一貫性レビュー）
     - STEP C: 設計レビュー（目的適合性・実用性・耐久性）
     - STEP R: リグレッションテスト（変更箇所の直接テスト + 影響範囲テスト）

5. バージョン末尾に「リリース前検証」セクションを記載する
   - リリース前検証で実施するSTEPの組み合わせを明記すること
   - マイナーバージョン: STEP A + STEP B + STEP R
   - メジャーバージョン: STEP A + STEP B + STEP C + STEP R

提案を以下の形式で報告すること（ROADMAP.md への書き込みはまだ行わない）：
- 各バージョンの枝番構成の一覧（枝番番号、対象課題ID、検証計画）
- 1枝番あたりの作業量の見積もり（軽い/普通/重い）
- バージョンの振り分け理由（なぜこの粒度で切ったか）
- 振り分けや順序で迷った判断があればその理由
- ユーザーに確認したい事項（「この機能は早めに欲しいか」等）

人間の確認が取れるまで ROADMAP.md の更新は行わないこと。
ISSUES.md は更新しないこと（ステータスの更新はフェーズ4で行う）。
```

**人間がやること:** AIの提案を確認し、修正指示があれば伝える。問題なければ「OK」と返す。

### プロンプト3-B：ROADMAP.md への書き込み（同じチャット内で続けて投げる）

```
確認が取れたので、提案内容を ROADMAP.md に反映すること。

- 既存の ROADMAP.md のフォーマット・記法に合わせること
- 先ほどの確認で修正指示があった場合は、それを反映した上で書き込むこと

更新完了後、変更箇所のサマリーを報告すること。
```

## フェーズ4：実装・検証

> **フェーズ4の全体像**
> ROADMAPの枝番を1つずつ消化していく。
> 1枝番につき「実装 → 検証 → 修正 → リグレッション＋完了処理」のサイクルを回す。

### フェーズ4 内のチャット構成

```
枝番ごとに以下を繰り返す：

チャット1: 実装（プロンプト4-A）
チャット2: STEP A 機能検証（プロンプト4-B）
チャット3: STEP B 品質レビュー（プロンプト4-B）← ROADMAPで指定されている場合のみ
チャット4: 修正 + リグレッション準備（プロンプト4-C → 4-D）← NGがあれば
チャット5: STEP R リグレッション + 完了処理（プロンプト4-E）
```

> **通常の枝番（STEP A + STEP R のみ）は最低3チャットで完了する。**
> STEP B が必要な場合でも最低4チャット。旧体系の5〜7チャットから大幅に削減。

**新しいチャットを開いて投げる。**
共通ヘッダー + 以下のSTEP R本体 の順で貼る（4-Bと同じ構造）。

> **Note**: リグレッションで全件OKの場合、同じチャット内で完了処理まで実行する。
> NGが見つかった場合はプロンプト4-Cに戻る（完了処理は行わない）。

```
【STEP R：リグレッションテスト】

WORK-LOG.md の「注意事項・申し送り」に記載されている【変更サマリー】に基づいて、
以下の2段階でリグレッションテストを行うこと。

■ 第1段階：変更箇所の直接テスト
変更した機能そのものが正常に動作するか確認する。

■ 第2段階：影響範囲の特定とテスト
1. 【変更サマリー】の「影響が想定される箇所」を起点に、コード上でさらに参照元を洗い出す
2. 洗い出した箇所の機能が変更前と同様に動作するか確認する
3. データの流れを追い、間接的に影響を受ける画面や処理があれば、それも確認する

報告形式：
| # | テスト区分（直接/影響範囲） | 機能名 | 操作内容 | 期待結果 | 実際の結果 | OK/NG |

コードの修正はしないこと。

結果はプロジェクトルート直下の qa-report.md の末尾に追記すること。
セクション見出しは「## STEP R：リグレッションテスト（vX.X.X 枝番N-M YYYY-MM-DD）」とすること。

【記載ルール】
このファイルは別チャットのAIが読んで修正作業を行う。
そのAIがこのファイルだけで修正箇所を特定できるレベルの詳細さで書くこと。
- NG項目には必ず以下をすべて含めること：
  - 該当ファイル名と行番号
  - 再現手順
  - 期待される挙動と実際の挙動の差分
  - 原因の推定
  - 今回の変更との因果関係（この変更が原因か、元から壊れていたか）
- OK項目は表の1行で十分。
- 影響範囲の洗い出し結果（どのファイル・関数を確認対象としたか）も記載すること。

■ リグレッション結果が全件OKの場合：完了処理

リグレッションテストで NG が0件だった場合、続けて以下の完了処理を行うこと。
NG が1件でもあった場合は完了処理を行わず、結果報告のみで終了すること。

1. ISSUES.md を読み、今回の枝番で対応した課題IDのステータスを 🟢 完了 に変更する
   - 対象課題IDは ROADMAP.md の該当枝番を参照して確認すること

2. ROADMAP.md の該当枝番に「✅ 完了」マークを付ける

3. WORK-LOG.md を更新する
   - 「最新の作業」を今回のリグレッション＋完了処理の内容で更新する
   - 過去の作業が3件を超える場合、古いものから削除すること
   - 「次にやるべきこと」に以下のいずれかを記載する：
     a. 同バージョン内に次の枝番がある場合 → 「枝番N-Mの実装に着手」
     b. 同バージョンの全枝番が完了した場合 → 「リリース前検証の実施」
     c. リリース前検証も完了している場合 → 「フェーズ5（動作確認）へ進む」

4. 更新内容を報告すること
```

### 補足：修正→検証の繰り返しが3回を超えた場合

修正とリグレッションを3回繰り返してもNGが解消しない場合は、
根本的な設計の問題がある可能性がある。

対処：
1. qa-report.md の残存NG項目を確認する
2. 残存NGの内容を課題メモとして、フェーズ2（課題の整理）に戻す
3. 該当の枝番は ROADMAP.md で「⏸️ 保留」とする
4. ISSUES.md に新規課題として登録し、次バージョン以降で対応を検討する

### 補足：リリース前検証

バージョンの全枝番が完了した後、ROADMAPの「リリース前検証」で指定されたSTEPを実施する。
プロンプトはプロンプト4-Bと同じ（人間がプロンプトを変更する必要はない。
枝番単位かリリース前かはAIが WORK-LOG.md と ROADMAP.md から自動判別する）。

リリース前検証でNGが出た場合は、プロンプト4-C → 4-D → 4-E の修正サイクルを回す。

#### STEP A のスコープ分割

リリース前の STEP A は全画面が対象のため、1チャットでは収まらない場合がある。
STEP A の検証スコープ指示に従い、AIが qa-report.md を読んで検証済み画面を自動判定するため、
人間は同じプロンプトを新しいチャットで繰り返し投げるだけでよい。
AIが「全画面検証済み」と報告したら次のSTEPに進む。

**分割不要なSTEP**: STEP B（品質レビュー）・STEP C（設計レビュー）は全体を俯瞰して
一貫性を判断するため、分割せず1チャットで実施する。

#### ビルド確認

リリース前検証がすべて完了した後、フェーズ5に進む前に `npm run tauri build` を実行して
ビルドが成功することを確認すること。ビルドエラーがある場合はフェーズ5に進まず、
エラーを修正してから再ビルドする。

> **Note**: この時点ではバージョン番号は更新しない（フェーズ7で更新する）。
> ビルド可否の確認が目的であり、成果物はリリースには使わない。

# 全体のクイックリファレンス

**「今何をすればいい？」がわからなくなったらここを見る。**

| 今の状況 | 次にやること | 投げるプロンプト |
|---|---|---|
| 使っていてバグや改善点を見つけた | メモする | なし（フェーズ1） |
| 課題メモが溜まった | AIに整理させる | プロンプト2-A → 確認 → 2-B |

| ロードマップができた | 最初の枝番の実装を始める | プロンプト4-A |
| 実装が終わった | 検証STEPを実行 | プロンプト4-B（STEP A） |
| STEP Aが終わった、STEP Bが必要 | STEP Bを実行する | プロンプト4-B（STEP B） |
| 検証STEPが全部終わった、NGなし | リグレッション＋完了処理 | プロンプト4-E |
| 検証STEPが全部終わった、NGあり | 修正する | プロンプト4-C → 4-D |
| 修正が終わった | リグレッション＋完了処理 | プロンプト4-E |
| リグレッションOK | 完了（4-E内で完了処理済み） | — |
| リグレッションNG | また修正する | プロンプト4-C に戻る |

| 実機確認OK | ビルドさせる | フェーズ7プロンプト |
| 実機確認で重大な問題 | 修正させる | プロンプト5-A |
| ビルドできた | 最終確認してリリース | なし（フェーズ8） |
| リリース完了 | 後片付けさせる | プロンプト8-A |

# バージョン別 実施STEPガイド

| バージョン種別 | 実施する検証STEP | 備考 |
|---|---|---|
| 枝番ごと（通常） | STEP A + STEP R | 基本セット（最低3チャット） |
| DB変更またはUI変更を伴う枝番 | STEP A + STEP B + STEP R | 品質レビュー追加（最低4チャット） |
| バージョン最終枝番 | STEP A + STEP B + STEP R | — |
| マイナーバージョン リリース前 | STEP A + STEP B + STEP R | フルセット |
| メジャーバージョン リリース前 | STEP A + STEP B + STEP C + STEP R | 設計レビュー追加 |
